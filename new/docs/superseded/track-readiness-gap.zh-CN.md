# 上赛道差距分析（基于现有代码）

## 1. 结论

当前工程状态更接近“板级 bring-up 基本打通，系统可在诊断模式下持续运行”，还不是“可以上赛道跑”。

从代码和最近板测证据看，当前距离“可上赛道”至少还缺四类关键能力：

1. 关键传感器闭环未完成，尤其是 IMU direct-match 仍未在当前可达板上完成验证。
2. 控制环虽然已经能启动，但大多数成功运行仍处于 `control.veto` 状态，不能证明车辆会稳定、可控地出力。
3. 视觉和参数链路仍明显依赖受控环境，尚未达到赛道环境所需的鲁棒性。
4. 执行器路径存在结构性缺口，尤其是转向执行链路是否完整仍未收口。

如果把“上赛道”拆成阶段目标，那么当前更像是停留在“L1: 板上程序能跑、日志可信、非 IMU 外设多数已打通”，距离“L3: 低速闭环可动”和“L4: 可在赛道环境中调参”还有明显差距。

## 2. 当前已经做到的事情

### 2.1 构建、部署、远程 smoke 基本可用

- `new/user/main.cpp` 已形成清晰的组合根，负责加载 profile、参数、启动平台、运行控制环和感知前端。
- `new/user/run_remote_smoke.sh` 已修复为“保留远端运行时退出码，不再被日志拷回结果掩盖”。
- 当前可达板地址已切换到 `10.100.170.226`，并且修复后二进制已完成重新部署。

### 2.2 非 IMU 的关键硬件路径已有实测进展

- 相机 direct-match 可以初始化并产出 `160x120 -> 160x128` 的适配帧。
- ADC 低压检测路径已经接到平台适配层，并在启动和运行时重复采样。
- 编码器桥接读路径已修复，不再把板端 `read() == 0` 的成功返回误判为失败。
- 电机桥接写路径已修复，不再把板端 `write() == 0` 的成功返回误判为失败。
- 定时器桥已经能拉起控制环，系统可以跑到 `LS2K_MAX_FRAMES` 设定上限后正常退出。

### 2.3 当前“成功运行”不等于“可上车闭环”

- 最近成功日志主要来自跳过 IMU 或同时跳过 IMU+编码器的降级运行：
  - `openspec/changes/fix-board-runtime-hardware-detection/verification/runtime-smoke-skip-imu-after-fix.log`
  - `openspec/changes/fix-board-runtime-hardware-detection/verification/runtime-smoke-skip-imu-encoder-after-fix.log`
- 这些日志证明系统能跑起来，但也同时证明控制环仍长期停留在 `control.veto`，因此它们不能作为“车已可跑”的证据。

## 3. 赛道就绪分级

建议把“上赛道”拆成以下五级状态：

| 级别 | 定义 | 当前判断 |
|---|---|---|
| L0 | 可编译、可部署、可回收日志 | 已达到 |
| L1 | 板上能稳定启动并持续运行，关键日志可信 | 基本达到 |
| L2 | 关键传感器有效，控制环可解除 veto，执行器能安全出力 | 未达到 |
| L3 | 低速闭环可动，可做直线、轻弯、停启测试 | 未达到 |
| L4 | 可在真实赛道环境下调参，能够适应不同光照和赛道材质 | 未达到 |
| L5 | 可稳定跑圈并具备比赛所需鲁棒性 | 远未达到 |

下面的未完成任务按“阻塞 L2/L3/L4”的方式列出。

## 4. 阻塞上赛道的未完成任务

### 4.1 IMU direct-match 仍未闭环

这是当前最明确、最直接的阻塞项。

现状：

- 默认硬件 profile 仍把 IMU 定义为 `direct-match`，见 `new/config/hardware_profile.json`。
- 启动阶段要求 IMU 是 direct-match 且 `Ready()`，否则启动失败，见 `new/code/runtime/startup.cpp`。
- 控制环阶段要求 `imu.valid` 为真，否则直接进入 `control.veto`，见 `new/code/runtime/control_loop.cpp`。
- `new/code/platform/imu_adapter.cpp` 和 `new/code/platform/true_ls2k0300/imu_bridge.cpp` 已经实现了设备枚举、路径探测和样本读取。
- 但当前 reviewer 仍将该 change 判为 `blocked`，原因是“尚未在当前可达板上做一次启用 IMU 的 direct-match 重跑”，见：
  - `openspec/changes/fix-board-runtime-hardware-detection/verification/implementation-local-review.json`
  - `openspec/changes/fix-board-runtime-hardware-detection/tasks.md`

为什么它阻塞上赛道：

- 现有转向控制直接使用 `gyro_z`，见 `new/code/legacy/pid_control.cpp`。
- 只要 IMU 没有真实有效样本，控制环就不会解除 veto。
- 这意味着车辆无法进入“真实姿态参与的闭环控制”。

未完成任务：

1. 在当前可达板上，使用启用 IMU 的目标 direct-match profile 重跑一次完整启动。
2. 记录 IMU 是“成功解析到真实节点”还是“明确失败且给出路径”，不能再停留在 unreachable 或 degraded-only 证据上。
3. 对 IMU 样本做基础真实性校验：
   - 静止时零偏是否可接受
   - `gyro_z` 符号方向是否正确
   - 样本是否连续、是否有明显跳变
4. 明确是否需要补充校准、滤波或坐标轴方向修正。

建议验收口径：

- 启动日志包含 `imu.init` 和 `imu.detect` 的可信 direct-match 结果。
- 控制环运行时 `imu.valid == true`，且不是偶发一次成功。
- 实车静止条件下，IMU 输出稳定，不出现持续漂移到不可用程度。

### 4.2 控制环目前多数时候仍处于 fail-safe veto

现状：

- `new/code/runtime/control_loop.cpp` 中，以下任一条件成立都会触发 veto：
  - 感知结果不 fresh
  - 感知结果自带 `emergency_veto`
  - 低压告警有效
  - `!imu.valid`
  - `!encoder.valid`
- 一旦 veto，控制器内部状态被冻结，输出命令被清成安全值。
- 最近的“成功运行”日志大量出现 `control.veto`，说明系统主要仍在“安全但不出力”的观察模式运行。

为什么它阻塞上赛道：

- 上赛道的最低要求不是“程序不崩”，而是“能够在传感器正常时稳定地解除 veto 并持续输出可控动作”。
- 现在的成功样例还没有证明这一点。

未完成任务：

1. 在 IMU 和编码器都有效时，确认 `control.veto` 能够退出，而不是永远保持常亮。
2. 增加一轮真实出力验证，确认 `ActuatorCommand` 不是始终保持 emergency-stop。
3. 验证 `state.actuators_armed` 能在满足条件时变为真，并在异常时及时拉回。
4. 补齐“从有效传感器到真实出力”的日志证据，而不是只看 startup 是否完成。

建议验收口径：

- 控制环在稳定数百个周期内存在持续的非 veto 状态。
- 电机应用路径存在真实的非零控制输出，而不是只有停机命令。
- 故障注入后可以重新回到 fail-safe。

### 4.3 差速转向拓扑已明确，但历史 servo 痕迹仍需收口

根据当前目标车体约束，车辆是“三轮车纯差速转向”，这意味着转向执行依赖左右轮差速，不依赖独立舵机。

现状：

- `new/code/legacy/motor_logic.cpp` 当前输出的是左右轮 PWM 差分，符合差速转向思路。
- `new/code/platform/motor_adapter.cpp` 也只实现了左右电机 PWM/GPIO 路径。
- 但 `port::ActuatorCommand` 里仍保留 `servo_pwm` 字段，见 `new/code/port/control_types.hpp`。
- `new/docs/mapping.md` 里仍保留了 `old/code/Servo.c` 的历史说明。

为什么它仍然是任务项：

- 这已经不再是“是否缺少舵机转向能力”的阻塞问题，而是“需要把目标执行器拓扑在代码和文档里明确收口”的一致性问题。
- 如果继续保留 `servo_pwm` 这类未使用字段而不解释，后续在控制、识别策略、参数设计中容易误导实现方向。

未完成任务：

1. 在文档中正式声明“当前目标平台采用纯差速转向，`servo` 不属于本阶段执行器链路”。
2. 评估是否移除或降级 `servo_pwm` 字段：
   - 如果短期无用，建议删除或标注为保留字段
   - 如果未来可能复用，至少要明确“当前运行时不消费该字段”
3. 审查控制链路命名，避免让 `turn_output` 语义被误解为“舵机角度”而不是“差速调节量”。

建议验收口径：

- 执行器拓扑在代码、文档、板测三处一致地表述为“纯差速转向”。
- 不再把 `servo` 作为赛道就绪的阻塞项，但要消除其历史残留造成的歧义。

### 4.4 视觉算法仍是“可工作”的基础版，不是“耐环境、可扩展”的赛道版

现状：

- 当前主感知算法在 `new/code/legacy/camera_logic.cpp`，核心是：
  - `OtsuThresholdFast`
  - `EightNeighborEquivalentError`
- 它本质上仍是单帧、全局阈值、加权中心误差类传统方法。
- 当前感知输出 `port::PerceptionResult` 仍偏向“循迹误差 + veto”这一类低维结果，尚未为“赛道图片识别/语义识别”预留明确扩展面。
- `PerceptionResult` 中虽然保留了 `circle_find` / `zebra_flag` / `cross_flag` 对应的 runtime 状态槽位，但当前新代码中并没有真正更新这些赛道元素状态：
  - `new/code/runtime/runtime_state.hpp` 有字段
  - 代码搜索只看到定义，没有真实驱动逻辑

为什么它阻塞上赛道：

- 这种算法在受控光照、单一赛道材质下可以工作，但在阴影、反光、曝光波动、局部脏污、线型变化下会明显退化。
- 当前实现还没有环岛、十字、斑马线等场景逻辑回收完成，也就谈不上比赛级赛道行为。
- 后续还计划增加“图像识别算法”，用于识别赛道上的不同图片并据此决定方向、速度等策略；但当前代码没有把这类高层语义输入与现有循迹输出解耦成清晰接口。

未完成任务：

1. 明确“赛道元素能力”的范围：
   - 只做基础循迹
   - 还是恢复原工程中的环岛、十字、斑马线等逻辑
2. 如果要恢复：
   - 把 `circle_find` / `zebra_flag` / `cross_flag` 从占位状态变成真实运行状态
   - 定义这些状态如何进入控制决策
3. 为未来的赛道图片识别算法预留接口，建议至少拆成两层：
   - 低层：现有循迹/边界/误差输出
   - 高层：图片识别或语义识别输出，例如“左转标志”“减速标志”“分叉决策标志”等
4. 明确高层识别结果如何影响控制决策：
   - 直接修改目标速度
   - 修改转向策略
   - 切换运行状态机
   - 触发临时减速、等待、转弯优先级等
5. 即使只做基础循迹，也需要补一轮鲁棒性增强：
   - 分区或局部阈值
   - 置信度输出
   - 丢线后的短时恢复策略
   - 更稳定的边界/中线提取，而不是只靠单帧加权中心

建议验收口径：

- 在至少几类典型光照和赛道材质下，感知输出的横向误差连续、可解释、不会大幅跳变。
- 感知层有“可用/不可用/低置信度”的明确状态，而不是只有硬阈值 veto。
- 未来图片识别算法可以在不重写现有控制环主干的前提下接入，只需向感知结果或策略层注入额外语义信号。

### 4.5 相机曝光和图像质量链路还不适合真实赛道环境

现状：

- `new/code/platform/camera_adapter.cpp` 对 direct-match 相机路径有明确限制：
  - 相机必须是 `160x120`
  - `exp_light` 如果不是默认值 `65`，则直接视为 unsupported 并 fail closed
- `new/code/platform/param_store.cpp` 也会在非默认 `exp_light` 时给出警告，说明当前 direct-match 路径不具备曝光调节支持。

为什么它阻塞上赛道：

- 赛道环境最常见的问题之一就是光照变化。
- 如果当前链路只能接受“默认曝光 + 受控场地”，那么算法再好也很难稳定跑圈。

未完成任务：

1. 确认当前相机和供应商公开 API 是否真的无法控制曝光。
2. 如果可以控制：
   - 接入新的 direct-match 或 adaptation-hook 链路
3. 如果不能控制：
   - 明确场地照明边界
   - 补充图像预处理以扩大可用环境范围
4. 做一轮图像采样，观察默认曝光在不同环境下的灰度分布和阈值稳定性。

建议验收口径：

- 在至少“室内均匀光”“局部阴影”“地面反光较强”三类条件下，图像仍可用于稳定感知。

### 4.6 编码器和速度闭环还缺少实车标定

现状：

- `new/code/platform/encoder_adapter.cpp` 已经实现：
  - first-sample baseline
  - 绝对计数差分
  - 大跳变阈值过滤
- 但当前阈值、计数方向、车轮速度换算都还是工程假设，还没有结合真实车体标定。

为什么它阻塞上赛道：

- 速度环是否稳定，取决于编码器方向、比例和噪声特性。
- 如果左右轮方向、脉冲比例或跳变阈值不合适，车在赛道上会表现为偏航、加减速不稳、启动冲击或刹停异常。

未完成任务：

1. 确认左右轮编码器方向与车辆前进方向一致。
2. 标定“计数差 -> 实际速度”的关系。
3. 重新评估 `20000` 这一 jump threshold 是否合理。
4. 在低速、匀速、刹停三种工况下验证编码器信号稳定性。

建议验收口径：

- 车辆匀速运行时，左右编码器差分与实际运动方向一致。
- 速度闭环不会因编码器噪声持续误判为 jump 或 invalid。

### 4.7 参数调参与赛道切换流程还没有形成工程闭环

现状：

- 现在参数是 JSON 文件加载，路径清晰，适合开发。
- 但缺少“赛道参数版本”“快速回滚”“环境组合配置”的成体系机制。
- 旧工程里依赖的菜单、在线调试、现场交互能力在 phase 1 中已被折叠掉。

为什么它阻塞上赛道：

- 真正上赛道时，不只是“代码对不对”，而是“参数调得动、改得快、错了能退回”。
- 只有单个 `default_params.json`，会让现场调参效率很低。

未完成任务：

1. 设计参数集管理方式，例如：
   - baseline
   - indoor-bright
   - indoor-shadow
   - track-a / track-b
2. 记录每次实测参数与效果，避免无序调参。
3. 明确哪些参数可以安全在线调整，哪些需要重启生效。

建议验收口径：

- 能在不改代码的前提下切换多组赛道参数。
- 出现退化时能快速回退到稳定参数集。

### 4.8 Linux 进程式运行时的时序稳定性尚未量化

现状：

- `new/user/main.cpp` 里，感知前端是前台循环 + `sleep_for(1ms)`。
- 控制环由 `TimerAdapter` 拉起，默认周期来自参数 `control_period_ms`，当前默认是 `5ms`。
- 现有代码可以运行，但还没有看到针对时序抖动、感知-控制耦合延迟的量化结果。

为什么它阻塞上赛道：

- 赛道表现很容易被调度抖动、图像处理耗时、感知发布时间漂移放大。
- 如果 perception 与 control 的时间关系不稳定，实车会出现输出忽快忽慢、转向抖动、滞后明显等问题。

未完成任务：

1. 增加关键时序观测：
   - 感知每帧耗时
   - 控制环周期抖动
   - perception 发布到 control 消费的延迟
2. 确认在 CPU 负载上升时是否还稳定。
3. 评估是否需要把感知和控制进一步解耦或增加缓冲策略。

建议验收口径：

- 控制环周期抖动在可接受范围内。
- 感知延迟不会持续导致 `stale`。

### 4.9 低压、异常恢复、停机链路仍需实车级验证

现状：

- 低压路径已经有 adapter 和日志，且 unavailable 会 fail-safe。
- shutdown 路径也会尝试停定时器、关电机、回收资源。
- 但这些大多仍停留在“代码路径存在”和“板测可跑”的阶段。

为什么它阻塞上赛道：

- 赛道上真正危险的不是正常工况，而是掉压、传感器瞬时失效、控制线程退出、急停后的恢复。

未完成任务：

1. 做低压阈值和误判率验证。
2. 验证运行中编码器/IMU 瞬断时是否能立即安全退回 veto。
3. 验证 shutdown / emergency-stop 后再次启动是否干净。

建议验收口径：

- 异常工况下始终优先进入 fail-safe。
- fail-safe 后不会留下执行器悬空状态。

## 5. 可以在没有 IMU 的情况下继续推进的部分

下面这些工作不需要等 IMU 完全恢复，可以并行推进：

1. 相机曝光链路和图像预处理增强。
2. 视觉算法鲁棒性增强。
3. 赛道元素状态机恢复或重构。
4. 编码器方向、比例、跳变阈值标定。
5. 参数集管理和调参流程建设。
6. 时序与性能观测埋点。
7. 把“纯差速转向”写成正式约束，并清理历史 servo 残留。

但是，这些工作只能提升“准备度”，不能替代 IMU direct-match 验证本身，也不能让当前 change 直接归档。

## 6. 建议的最短推进路径

如果目标是“尽快从现在推进到能上车、再推进到能上赛道”，建议采用下面的顺序。

### 第一阶段：达到“低速可动”的最低条件

1. 把纯差速转向约束固化到文档和代码接口，移除历史歧义。
2. 完成 IMU direct-match 板测闭环。
3. 让控制环在真实传感器有效时解除 veto。
4. 做低速、短距离、可人工接管的实车出力测试。

### 第二阶段：达到“可调参”的最低条件

1. 稳定编码器和速度闭环。
2. 补相机曝光或图像预处理。
3. 为未来图片识别算法建立独立接口层，不把语义识别直接塞进现有循迹实现里。
4. 增加时序、延迟、置信度观测。
5. 建立多套参数配置。

### 第三阶段：达到“可上赛道试跑”的最低条件

1. 恢复或替代必要的赛道元素逻辑。
2. 做多环境图像测试。
3. 做连续多圈稳定性测试。
4. 固化一套可回滚的赛道参数和启动流程。

## 7. 当前最值得先确认的三个问题

为了避免无效投入，建议先回答下面三个问题：

1. 纯差速转向在代码接口层是否现在就要正式固化，而不是继续保留历史 servo 痕迹？
2. IMU 硬件问题是“暂时没通”，还是“这个阶段打算长期绕开”？
3. 赛道图片识别算法希望作为独立语义模块接入，还是与现有循迹逻辑深度耦合？

这三个问题的答案会直接决定后续任务顺序，以及哪些工作是主线、哪些工作只是优化项。
