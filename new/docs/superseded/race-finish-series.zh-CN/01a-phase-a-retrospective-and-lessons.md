# Phase A 附录：补充经验与后续借鉴

## 1. 文档定位

这份附录只记录一类内容：

- `Phase A` 主文档、路线图、当前进展中没有系统展开
- 但对后续阶段和同类项目明显有借鉴价值

因此，它不是 `Phase A` 的重复摘要，也不是另一个验收结论页。

## 2. 这次过程额外证明的重要经验

### 2.1 先确认“板上跑的是不是你以为的那个版本”

硬件验证里，一个很容易被忽视的问题是：

日志看起来对了，不等于板上二进制就是本地刚改完的版本。

后续应把这条当成固定动作：

1. 上传后先确认板端二进制标识
2. 再跑验证
3. 只对“版本已确认”的日志下结论

这条规则的泛用价值很高，因为它能直接避免把旧版本行为误判成新改动结果。

### 2.2 访问链路本身也属于验证对象

这次过程里，SSH 并不是始终稳定可用，串口一度成为继续推进的必要手段。

这说明硬件阶段不应默认：

1. SSH 永远可达
2. 网络问题和系统问题可以完全分离
3. 远程入口失败就等于运行时失败

更稳妥的做法是始终保留一条备用访问链路，例如：

1. SSH
2. 串口
3. 本机绑定设备节点

后续阶段只要还依赖真板，就应把“链路退化时如何继续验证”视为流程的一部分。

人工备注：串口在Windows端永远可达，可以自行绑定到WSL端。再通过串口查询IP连接SSH

### 2.3 部署成功和验证成功要分开记录

这次过程说明，“已上传”“可运行”“已通过验证”是三件不同的事。

后续更稳妥的做法是分别记录：

1. 部署是否成功
2. 启动是否成功
3. 验证目标是否达成

这样当问题发生时，可以快速判断故障是在：

1. 部署链路
2. 启动链路
3. 运行时语义
4. 阶段出口判断

### 2.4 半截日志和失效证据要及时清理

硬件调试里最危险的噪音之一，不是没有日志，而是留下了：

1. 中途失败的日志
2. 语义已经过时的日志
3. 名字像正式证据、实际上并未完成的日志

这类文件会持续污染后续判断。

因此后续应形成固定习惯：

1. 失败产物要标明失败
2. 半截产物要么删除，要么明确标注无效
3. 只把真正可引用的文件留在正式证据路径

### 2.5 单板验证要有“唯一权威结论路径”

这次过程里，真正能稳定收口的，不是零散聊天式结论，而是把最终判断回填到统一文档位置。

这说明后续阶段应该维持：

1. 现场观察可以很多
2. 临时日志可以很多
3. 但权威结论只能回到少数固定文档

否则后面接手的人会面对多个彼此竞争的“最新判断”。

### 2.6 阶段结束和实现审查结束，不是同一个概念

这次过程里，一个容易混淆的点是：

1. 阶段退出条件是否满足
2. 某个 change 的 review/archive 流程是否完全结束

这两者相关，但不完全同义。

后续推进时，应避免：

1. 因为流程工件还在补，就否认已经完成的阶段事实
2. 因为阶段已通过，就跳过实现审查和归档要求

更好的做法是把“工程事实”和“流程状态”分开写清楚。

### 2.7 人工参与型测试也要当作正式测试设计

这次 IMU 方向与静止样本取证，实际上包含了明确的人工作用。

这类测试如果没有被命名和约束，很容易退化成：

1. 操作者知道自己做了什么
2. 后来的人不知道当时到底怎么做的

后续应把人工参与型测试也写成最小测试设计：

1. 操作对象是什么
2. 操作方向是什么
3. 操作持续多久
4. 期待看到哪类响应

这样它才具备复现价值。

### 2.8 低风险配置本身也要纳入证据管理

这次过程里，低风险参数不是单纯的临时手段，而是阶段收口的重要组成部分。

这说明后续阶段应避免把低风险配置当成“随手改一下”的隐式背景。

更稳妥的做法是：

1. 给低风险配置单独命名
2. 保存其来源和目标
3. 说明它回答的验证问题是什么
4. 明确它不是比赛默认配置

这样后续就不会把“验证配置”误读成“正式运行配置”。

### 2.9 “不影响后续”本身就是一种设计目标

这次很多测试之所以可接受，不只是因为测到了东西，而是因为它们满足：

1. 默认关闭
2. 不改生产主路径语义
3. 不需要长期保留临时耦合
4. 测完不会留下隐藏状态

这说明以后增加 bench 工具、诊断入口、临时 profile 时，应该把“可撤销性”当成一等目标。

### 2.10 单次测试最好只跨一个解释边界

这次过程反复证明，一个测试如果同时跨越：

1. 网络链路
2. 部署链路
3. 设备语义
4. 控制逻辑
5. 执行器出力

那它即使失败，也很难快速判断失败层级。

因此后续更好的设计是让一次测试尽量只跨一个新的解释边界，其余部分尽量保持已知稳定。

### 2.11 文档同步必须成批完成，不能只改一处

阶段状态一旦变化，通常受影响的不止一个文件。

这次过程说明，至少要同步检查：

1. 当前阶段文档
2. 总路线图
3. 当前进展页
4. 阶段结论证据页
5. README 或入口索引

只改一处，其他地方保留旧判断，后面几乎一定会重新引发争议。

### 2.12 历史材料多的时候，最需要的是“当前 authoritative path”

这次工作同时涉及：

1. 主系列文档
2. superseded 文档
3. 归档 change
4. 临时日志
5. 历史代码分支

信息很多时，真正关键的不是“把所有材料都看一遍”，而是先明确：

1. 哪个路径是当前主结论
2. 哪个路径只保留历史背景
3. 哪个路径只是现场临时产物

没有这个区分，接手者很容易在旧结论里兜圈子。

### 2.13 硬件验证要把“人身安全”和“结论有效性”同时优化

这次过程里，安全边界和验证质量并不是天然冲突的。

反而很多高质量证据，恰恰是在更安全的前提下拿到的，例如：

1. diagnostics-only 默认入口
2. env-gated bench PWM
3. 受限参数下的短窗口验证

这说明后续不应把“更安全”理解成“更保守、更没价值”，而应该把它理解成“更可重复、更可解释”。

### 2.14 真正省时间的不是“少写文档”，而是少留歧义

这次过程的一个反直觉经验是：

最费时间的往往不是缺文档，而是文档、日志、代码、板端现实之间存在轻度错位。

因此后续最值得坚持的不是“文档最少化”，而是：

1. 让结论足够短
2. 让路径足够明确
3. 让状态足够一致

只要这三点满足，文档就不会成为负担，反而会显著降低返工成本。

### 2.15 阶段切换前，最好做一次“切换用验证”

这次正式结束 `Phase A` 之前，再做一次板端验证是有价值的。

它的作用不只是“再测一次”，而是确认：

1. 当前板端仍可达
2. 当前二进制仍是预期版本
3. 当前默认入口仍符合收口后的安全语义
4. 阶段切换不是建立在过时环境之上

这类“切换用验证”后续在 `Phase B -> C`、`C -> D` 也值得保留。

## 3. 对后续阶段的直接要求

基于以上补充经验，后续阶段建议新增以下固定动作：

1. 每次板端关键验证前，先确认部署版本标识
2. 每个阶段都预留备用访问链路，不把 SSH 当成唯一入口
3. 对人工参与型测试写最小操作说明
4. 将部署成功、启动成功、验证成功分开记录
5. 阶段状态变化后，一次性同步所有主文档
6. 正式证据目录中不保留语义不明的半截日志
7. 低风险配置要单独命名并保存用途说明
8. 阶段切换前做一次简短的切换用验证

## 4. 一句话结论

这次 `Phase A` 除了完成硬件闭环，还额外证明了一件事：

在真板项目里，访问链路、版本确认、证据清洁度、文档同步和测试可撤销性，和算法本身一样重要。
